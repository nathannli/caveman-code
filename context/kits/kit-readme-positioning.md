# Kit: README Positioning
**Domain:** readme-positioning
**Version:** 1.0.0
**Status:** draft

## Requirements

### R-001: Clear product promise
README must make product identity and primary value understandable within the opening screen. A new reader must quickly understand what Cave is, who it is for, and why it is distinct.

**Acceptance Criteria:**
- AC-1: Before the first major scroll break, README states product category, intended user, and primary differentiator in plain language.
- AC-2: Opening headline and supporting copy can be understood without prior knowledge of Cave, Pi, or repository history.
- AC-3: Opening section identifies at least 1 concrete workflow or outcome the product enables.

### R-002: Problem-first framing
README must explain the user problem Cave solves before asking readers to evaluate detailed features. Benefits must be tied to recognizable user pain rather than feature inventory alone.

**Acceptance Criteria:**
- AC-1: README describes at least 1 common developer workflow or frustration the product improves.
- AC-2: Core benefits are expressed in terms of observable user outcomes rather than implementation details.
- AC-3: Top-level narrative emphasizes no more than 3 primary product messages so readers can retain the pitch after a quick scan.

### R-003: Audience fit and use-case clarity
README must help readers decide whether the product matches their needs. It must communicate expected user profile, assumptions, and representative use cases.

**Acceptance Criteria:**
- AC-1: README includes at least 3 concrete example use cases or tasks the product supports.
- AC-2: README states prerequisites or assumptions that affect who can successfully use the product.
- AC-3: README makes clear whether it targets end users, extension authors, or both.

## Out of Scope
- Full product tutorial beyond first-use orientation
- Exhaustive explanation of internal architecture
- Package-by-package API reference
